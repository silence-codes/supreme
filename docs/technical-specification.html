Обзор проекта и цели рефакторинга

Текущее состояние (v1.x)

Supreme Security — расширение для VS Code, выполняющее локальный анализ кода на уязвимости с использованием движка Trivy. Перед сканированием пользователь обновляет базу данных угроз через кнопку «Update Database». Результаты отображаются в боковой панели, древовидном представлении и HTML-отчётах.

  • Один тарифный план, одна лицензия
  • Движок: Trivy (CLI, embedded binary)
  • Threat Intelligence: обновление БД через Trivy DB + собственный API
  • Отчёты: HTML-панель, JSON-экспорт, CodeLens, диагностики

Цель рефакторинга (v2.0)

Разделить расширение на два тарифных плана с разными возможностями и бэкендом, сохранив единый пользовательский интерфейс и кодовую базу.

  • Supreme Lite — бесплатный / базовый тариф на open-source движках
  • Supreme Max — premium-тариф с проприетарным Threat Intelligence и интеграцией LLM через MCP Server
  • Единая кодовая база с feature-flags на уровне подписки
  • Сохранение текущего UI расширения без изменений

Бэкенд: Замена Trivy на мульти-движковую архитектуру

4.1 Интерфейс IScanEngine

Каждый движок реализует единый контракт. Это позволяет добавлять новые движки без изменения UI и оркестратора.

interface IScanEngine { /** Уникальное имя движка */ readonly name: string; /** Проверить наличие и установить CLI-бинарник */ ensureInstalled(): Promise<void>; /** Запустить сканирование целевой директории */ scan(targetPath: string, options: ScanOptions): Promise<ScanResult[]>; /** Отменить текущее сканирование */ cancel(): void; /** Текущее состояние */ isRunning(): boolean; } interface ScanOptions { excludePaths: string[]; severityFilter: SeverityLevel[]; timeout: number; cancellationToken: CancellationToken; }

4.2 Реализации движков

Движок Тип анализа Установка Формат вывода Тарифы
OSV-Scanner SCA — уязвимости зависимостей Go binary, GitHub Releases JSON (--format json) Lite Max
Semgrep SAST — паттерны в коде pip / standalone binary JSON (--json) Lite Max
Gitleaks Secrets — ключи, токены, пароли Go binary, GitHub Releases JSON (--report-format json) Lite Max
Checkov IaC — Dockerfile, Terraform, K8s pip / standalone binary JSON (--output-format json) Lite Max

4.3 ScanOrchestrator — оркестратор сканирования

Заменяет текущий TrivyService. Управляет жизненным циклом всех движков, агрегирует результаты в единый формат ScanResult[].

class ScanOrchestrator { private engines: IScanEngine[]; private tierManager: TierManager; async runFullScan(target: string): Promise<ScanResult[]> { const activeEngines = this.tierManager.getEnabledEngines(); const results = await Promise.allSettled( activeEngines.map(e => e.scan(target, this.buildOptions())) ); return this.mergeResults(results); } cancelAll(): void { this.engines.forEach(e => e.cancel()); } }

4.4 Унификация результатов

Каждый движок возвращает сырой JSON. Для каждого движка реализуется адаптер-парсер, преобразующий результаты в UnifiedIssue:

// Существующий интерфейс остаётся как есть interface UnifiedIssue { ID: string; // CVE-2024-XXXX, semgrep-rule-id, gitleaks-id Name: string; Severity: 'CRITICAL' | 'HIGH' | 'MEDIUM' | 'LOW'; Description: string; Type: 'Vulnerability' | 'Misconfiguration' | 'Secret' | 'CodeIssue'; Source: string; // 'osv-scanner' | 'semgrep' | 'gitleaks' | 'checkov' FixedVersion?: string; StartLine?: number; EndLine?: number; CodeSnippet?: string; Remediable?: boolean; // true если LLM может исправить (Max only) }

Threat Intelligence: Lite vs Max

5.1 Lite — Open-Source Threat Intelligence

Кнопка «Update Database» в sidebar обновляет локальные БД всех движков из публичных источников:

  • OSV-Scanner: автоматически скачивает из osv.dev при каждом запуске (встроенное поведение)
  • Semgrep: обновление набора правил через semgrep --config auto (semgrep registry)
  • Gitleaks: обновление конфигурации через GitHub Releases (gitleaks.toml)
  • Checkov: встроенное обновление правил при запуске

5.2 Max — Проприетарный Supreme Threat Intelligence

В дополнение к open-source фидам, Max-подписчики получают данные из Supreme TI Server:

  • Endpoint: GET /api/ti/feed?since={timestamp}
  • Содержит: Zero-day advisories, ранние предупреждения, EPSS-скоринг, exploit maturity
  • Формат: JSON-массив расширенных advisory-объектов
  • Кэширование: Локальный кэш в globalStorage, инкрементальные обновления
  • Обогащение: После сканирования движками, результаты обогащаются данными из TI (correlate по CVE-ID)
// Supreme TI обогащает результаты сканирования class ThreatIntelService { async enrichResults(issues: UnifiedIssue[]): Promise<EnrichedIssue[]> { const feed = await this.fetchLatestFeed(); return issues.map(issue => ({ ...issue, epssScore: feed.getEPSS(issue.ID), exploitMaturity: feed.getExploitStatus(issue.ID), supremeAdvisory: feed.getAdvisory(issue.ID), priorityScore: this.calculatePriority(issue, feed), })); } }
6

MCP Server и LLM Remediation (Max only)

⚠️
Ключевое различие: Lite — генерирует отчёты и информирует. Max — действует: через MCP Server интегрируется с LLM-агентами и автоматически исправляет обнаруженные уязвимости.

6.1 MCP Server — Model Context Protocol

Supreme Max регистрирует себя как MCP-сервер, предоставляя LLM-агентам (GitHub Copilot, Claude, и др.) набор инструментов для работы с результатами сканирования.

// MCP Tools, доступные LLM через Supreme Max const mcpTools = { "supreme.getVulnerabilities": { description: "Получить список уязвимостей из последнего скана", params: { severity?: SeverityLevel, type?: IssueType }, returns: UnifiedIssue[] }, "supreme.getIssueDetails": { description: "Получить подробности уязвимости с code snippet", params: { issueId: string }, returns: EnrichedIssue }, "supreme.applyFix": { description: "Применить исправление к файлу", params: { filePath: string, fix: CodeFix }, returns: { success: boolean, diff: string } }, "supreme.rescan": { description: "Перезапустить сканирование после исправлений", params: { target?: string }, returns: ScanResult[] }, "supreme.getRemediationPlan": { description: "Получить план исправления для конкретной уязвимости", params: { issueId: string }, returns: RemediationPlan } };

6.2 LLM Remediation Pipeline

Workflow автоматического исправления уязвимостей через LLM:

  • Шаг 1: Пользователь нажимает «Auto-Fix» на уязвимости в отчёте
  • Шаг 2: Supreme MCP Server отправляет контекст уязвимости LLM-агенту: code snippet, описание CVE, рекомендации из TI
  • Шаг 3: LLM генерирует патч (diff) для исправления
  • Шаг 4: Пользователь видит предложенные изменения в VS Code diff-view
  • Шаг 5: После применения — автоматический ре-скан для верификации
🔒
Безопасность: LLM-генерированные исправления никогда не применяются автоматически. Пользователь всегда видит diff и подтверждает изменения вручную.

6.3 MCP Server — техническая реализация

// Регистрация MCP Server при активации расширения (Max only) if (tierManager.isMax()) { const mcpServer = new SupremeMCPServer(scanOrchestrator, tiService); mcpServer.registerTools([ new GetVulnerabilitiesTool(), new GetIssueDetailsTool(), new ApplyFixTool(), new RescanTool(), new GetRemediationPlanTool(), ]); mcpServer.start(); // stdio transport context.subscriptions.push(mcpServer); }

TierManager — управление подписками

7.1 Логика определения тарифа

enum Tier { LITE = 'lite', MAX = 'max' } class TierManager { private currentTier: Tier = Tier.LITE; /** Валидация подписки через бэкенд */ async validateSubscription(licenseKey: string): Promise<Tier> { const response = await api.post('/api/license/validate', { key: licenseKey }); this.currentTier = response.data.tier; // 'lite' | 'max' return this.currentTier; } isMax(): boolean { return this.currentTier === Tier.MAX; } isLite(): boolean { return this.currentTier === Tier.LITE; } /** Какие движки доступны на текущем тарифе */ getEnabledEngines(): IScanEngine[] { const base = [osvScanner, semgrep, gitleaks, checkov]; if (this.isMax()) { return [...base]; // + расширенные правила } return base; } /** Какие фичи доступны */ canUseMCP(): boolean { return this.isMax(); } canUseAutoFix(): boolean { return this.isMax(); } canUseThreatIntel(): boolean { return this.isMax(); } }

7.2 Изменения в бэкенде (API)

Endpoint POST /api/license/validate должен возвращать расширенную информацию:

// Текущий ответ { "valid": true, "expires": "2027-01-01" } // Новый ответ v2.0 { "valid": true, "tier": "max", // 'lite' | 'max' "expires": "2027-01-01", "features": { "mcp": true, "threatIntel": true, "autoFix": true, "maxEngines": true } }

План миграции: от Trivy к мульти-движкам

Принципы миграции

  • Нулевые изменения в UI: sidebar, tree view, CodeLens, отчёты — всё работает как раньше
  • Обратная совместимость: существующая история сканов (globalState) сохраняется
  • Постепенный переход: можно оставить Trivy как fallback-движок на переходный период
  • Feature-parity: Lite v2.0 должен покрывать ≥100% функциональности текущей v1.x

Критерии приёмки

Lite

  • ✅ Сканирование проекта запускается без Trivy, используя OSV + Semgrep + Gitleaks
  • ✅ Результаты отображаются в sidebar, tree view и отчётах (как в v1.x)
  • ✅ CodeLens-маркеры работают для всех типов issues
  • ✅ Кнопка «Update Database» обновляет все open-source фиды
  • ✅ Отмена сканирования корректно останавливает все движки
  • ✅ CLI-инструмент работает с новыми движками
  • ✅ Первый запуск: автоустановка всех необходимых бинарников с прогрессом

Max

  • ✅ Все критерии Lite выполнены
  • ✅ Threat Intelligence обогащает результаты данными из Supreme TI Server
  • ✅ MCP Server запускается и отвечает на вызовы от LLM-агентов
  • ✅ LLM может получить список уязвимостей через MCP tool
  • ✅ LLM может сгенерировать и предложить исправление через diff-view
  • ✅ Ре-скан после исправления подтверждает устранение уязвимости
  • ✅ Подписка Lite не активирует MCP/TI функции (feature gate работает)
